# FastPick Debug输出优化计划

## 📋 一、现状分析

### 1.1 当前问题
- **分散混乱**: Debug输出分散在14个文件中，没有统一管理
- **格式不统一**: 多种前缀格式混用
  - `[智能预加载]`、`[分级预取]`、`[缓存命中]`
  - `[高分辨率]`、`[EXIF方向]`、`[分辨率自适应]`
  - 还有一些没有前缀的输出
- **冗余过多**: 大量重复的流程性输出
- **性能展示零散**: PerformanceMonitorService已有，但未充分整合
- **缺少级别控制**: 无法动态调整日志详细程度

### 1.2 涉及文件统计
- `Views/MainPage.xaml.cs`: 约80+处Debug输出
- `Services/PreviewImageService.cs`: 约30+处Debug输出
- `Services/ThumbnailService.cs`: 约10+处Debug输出
- 其他11个服务文件: 若干处输出

## 🎯 二、优化目标

### 2.1 核心目标
1. **统一化**: 创建专业的日志系统，统一输出格式
2. **级别化**: 支持多种日志级别，可动态控制
3. **性能化**: 整合性能监控，提供专业的性能指标展示
4. **精简化**: 剔除冗余输出，保留关键信息
5. **可配置**: 支持通过设置控制日志输出

### 2.2 具体指标
- 减少Debug输出数量: 60%+
- 性能关键路径全覆盖
- 支持4种日志级别
- 集成性能统计展示

## 💡 七、最佳实践

### 7.1 日志编写原则
1. **INFO级别**: 业务流程关键节点、重要决策
2. **WARN级别**: 可恢复的失败、降级处理
3. **ERROR级别**: 不可恢复的错误、异常
4. **PERF级别**: 性能关键操作、耗时统计
5. **VERBOSE级别**: 详细调试信息（默认关闭）

### 7.2 性能数据规范
- 关键操作必须记录耗时
- 使用 `using (logger.StartTimer("操作名"))` 模式
- 输出格式: `操作名: Xms`

### 7.3 避免的陷阱
- ❌ 不要在循环中大量输出日志
- ❌ 不要输出敏感信息（文件路径可以）
- ❌ 不要只输出 "进入/退出" 方法
- ✅ 要输出 "做了什么" + "结果如何"
