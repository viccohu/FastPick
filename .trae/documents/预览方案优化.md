# 一、方案中存在的核心问题（按危险等级排序）

## 🔥 危险等级 1：线程安全问题（必崩溃）

### 问题

你在 PreviewImage、PreviewCache 中**没有任何线程安全设计**。

预览加载、预解码、UI 切换、后台任务全部是**多线程并发**的。

会出现：

- 多个线程同时给 `item.PreviewCache` 赋值
- LRU 缓存并发修改
- 任务重复启动
- **UI 线程访问非线程安全集合 → 直接崩溃**

### 你的代码隐患

csharp

运行

```
public PreviewImage? PreviewCache { get; set; } // 非线程安全
public CancellationTokenSource? PreviewLoadCts { get; set; } // 并发覆盖

```

多个后台任务同时修改，会导致**令牌丢失、加载混乱、缓存覆盖**。

***

## 🔥 危险等级 2：CancellationToken 管理错误（会泄漏 + 卡死）

### 问题

你设计了 `PreviewLoadCts`，但**没有处理并发竞争**。

场景：

1. 快速翻页 → 取消上一个任务
2. 新任务还没创建完，旧令牌被覆盖
3. **任务永远无法取消 → 内存泄漏 + CPU 占用飙升**

### 根本原因

- 赋值 `PreviewLoadCts = new CTS()` 不是原子操作
- 多线程下会被覆盖
- 无法安全取消上一个任务

***

## 🔥 危险等级 3：双缓冲 UI 没有同步锁（会闪烁、黑屏、图片错乱）

### 问题

`_frontImage` / `_backImage` 切换时：

- 后台加载完成
- UI 同时渲染
- **跨线程操作 UI 控件 → 异常 / 闪烁 / 黑屏**

WinUI 的 `Image` 控件**必须在 UI 线程设置 Source**。

你当前方案**没有 DispatcherQueue 同步**。

***

## 🔥 危险等级 4：三级加载缺少状态锁（会重复加载、浪费性能）

### 问题

`PreviewLoadState` 没有线程安全保护：

1. 线程 A 检查状态 = NotLoaded
2. 线程 B 同时检查状态 = NotLoaded
3. **两个线程同时开始解码 → 双倍内存、双倍 CPU**

这是**预览模块最常见的性能黑洞**。

***

## 🔥 危险等级 5：预解码窗口没有 “取消远图” 机制（快速翻页会卡）

### 问题

你设计了 `PreDecodeAroundAsync(前后2张)`，

但**没有取消超出窗口的预解码任务**。

快速滑动时：

- 大量远图任务还在跑
- CPU/IO 被占满
- **当前图加载延迟**

***

## 🔥 危险等级 6：RAW 内置预览解码没有异常防护（会崩溃）

### 问题

你的代码：

csharp

运行

```
var previewStream = await decoder.GetPreviewAsync();

```

很多损坏的 RAW、旧相机 RAW **不包含 Preview**，会直接抛出异常导致**整个预览失败**。

你虽然加了 try-catch，但**粒度不够**，仍会崩。

***

## 🔥 危险等级 7：缺少内存过载保护（预览大图必爆内存）

### 问题

预览图是**高清大图**，比缩略图大 **30\~100 倍**。

你只写了 LRU，但**没有强制内存上限**。

浏览 50 张原图 → **内存轻松突破 1GB\~2GB → 应用被系统杀死**。

***

## 🔥 危险等级 8：优先级调度器没有全局单例（多任务混乱）

### 问题

你设计了 `PriorityTaskScheduler`，

但**没有设计成单例**。

会导致：

- 多个调度器同时运行
- 任务无法统一取消
- 优先级混乱
- 并发不可控

***

# 二、除了隐患，还有 4 个设计可大幅增强体验（非 bug，但建议优化）

1. **快速翻页时应直接停在 L1或L2，不加载 L3**
2. **预解码应判断 CPU 负载，不卡主线程**
3. **缓存应区分 “预览缓存” 和 “缩略图缓存”，互不干扰**
