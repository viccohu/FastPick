对于**大图预览区（Detail/Main Preview）**，在大批量、双路径（RAW+JPG）场景下，它的挑战在于：**如何在用户快速切换缩略图时，实现“瞬时”加载，同时保证 RAW 文件的渲染质量。**

以下是针对大图预览区的“优秀方案”实现细节：

***

### 1. 核心分层架构 (Layered Rendering)

大图区域不应该只是一张 `Image`。为了极致的流畅度，建议采用**三层金字塔加载模式**：

- **第一层：占位层 (Placeholder)**
  - 直接显示该资产已有的**缩略图**（从 `ThumbnailRepeater` 的缓存中取）。
  - **目的**：用户点击瞬间立刻有画面，消除白屏感。
- **第二层：快速预览层 (Fast Preview)**
  - 从文件读取内嵌的 **Screen-Size Preview**。
  - **技巧**：JPG 直接读原图（设定 `DecodePixelWidth` 为屏幕宽度）；RAW 文件调用 `GetThumbnailAsync` 且尺寸设为 `1024` 或以上，这通常是读内嵌的高清预览图，而非解析 RAW 数据。
- **第三层：原片解码层 (Full Decode)**
  - 仅在用户停止操作（如 500ms 后）且需要查看细节时，才进行全分辨率解码。

### 4. 关键逻辑实现 (C# Logic)

#### A. “预读”逻辑 (Pre-fetching)

当用户选中第 `N` 张图时，后台应自动开始加载 `N-1` 和 `N+1` 的预览图到内存缓存中。

C#

```
private async Task UpdateMainPreviewAsync(AssetPair selectedItem)
{
    // 1. 立即显示缩略图作为占位
    MainPreviewImage.Source = selectedItem.Thumbnail;

    // 2. 异步加载高清预览
    var highResSource = await Task.Run(async () => {
        // 优先尝试从 JPG 加载，因为解码速度比 RAW 快 10 倍
        var file = selectedItem.JpgFile ?? selectedItem.RawFile;
        
        var bitmap = new BitmapImage();
        using (var stream = await file.OpenReadAsync())
        {
            // 关键：限制解码尺寸为显示器宽度，避免浪费内存
            bitmap.DecodePixelWidth = (int)MainPreviewContainer.ActualWidth;
            await bitmap.SetSourceAsync(stream);
        }
        return bitmap;
    });

    MainPreviewImage.Source = highResSource;
}

```

#### B. RAW 文件的特殊处理

如果用户想看 RAW 的真实色彩（而非内嵌 JPG 预览），需要引入 `BitmapDecoder`：

- **WIC 模式**：WinUI 3 原生支持 WIC。通过 `BitmapDecoder.CreateAsync(stream)` 可以获取图像。
- **侧边栏 XMP 联动**：在大图预览时，实时读取对应的 `.xmp` 文件。如果用户修改了评分，立即更新 UI 并准备写入 XMP。

### 5. 性能优化的“天花板”细节

1. **利用共享内存 (Shared Surface)**：如果可能，让缩略图和大图共用同一块内存缓冲区，减少拷贝。
2. **GPU 加速渲染**：确保 `CanvasDevice` 开启。在 WinUI 3 中，`Image` 控件已经是硬加速的，但如果你要实现“100% 缩放下的丝滑平移”，建议研究 `ScrollViewer` 嵌套 `Canvas` 的模式。
3. **双进程/多线程解耦**：将“文件扫描”和“图片解码”放在两个独立的 `TaskScheduler` 中，防止扫描新文件夹时导致大图预览卡顿。

### **4. 极致效率的“黑科技”处理**

#### **A. 解码尺寸控制 (Decoding Scale)**

**绝对不要直接解码原图。** 如果用户的屏幕是 2K (2560px)，那么解码时设置 `DecodePixelWidth = 2560`。这样即使是 6000 万像素的 RAW 文件，内存占用也会缩减到原本的 1/10 左右。

#### **B. 利用 WIC 的** `GetPreview` **(针对 RAW)**

RAW 文件的完整渲染（Demosaic）极其耗电且慢。

- **优秀做法**：大图切换时，先调用 `BitmapDecoder.GetPreviewAsync()`。
- **效果**：绝大多数相机（索尼、佳能、尼康）都会在 RAW 里塞一张全尺寸或半尺寸的 JPG 预览。这张图的读取速度是秒级的，质量足以进行“选片”参考。

#### **C. 任务取消机制 (CancellationToken)**

如果用户快速按方向键（啪啪啪连按），你必须立即取消掉之前正在进行的解码任务。

- 每次切换时，调用 `_cts.Cancel()` 并创建新的 `CancellationToken`。
- 这能保证 CPU 始终在处理用户当前看到的这一张，而不是在后台死磕已经跳过的图。

## **2. 核心技术要点 (Technical Requirements)**

### **2.1 智能解码尺寸限制 (Critical)**

**严禁原图解码**。WinUI 3 渲染 6000 万像素原图会迅速耗尽显存。

- **执行逻辑**：获取当前显示器分辨率（或 Preview 容器的 `ActualWidth`）。
- **参数设定**：设置 `BitmapImage.DecodePixelWidth = (int)container.Width * DeviceFamily.RasterizationScale`。
- **效果**：内存占用从 \~200MB/张 降低至 \~15MB/张。

### **2.2 RAW 文件的“双通道”预览**

针对 RAW 文件（如 .ARW, .CR2），解析 Demosaic 数据极慢。

- **通道 A (瞬时)**：调用 `BitmapDecoder.GetPreviewAsync()`。这是相机内嵌的 Full-size JPG，速度比解码 RAW 快 20 倍。
- **通道 B (深度)**：仅在用户缩放至 100% 或停留超过 2 秒时，才启动深度 RAW 转换引擎。

### **2.3 任务撤销机制 (CancellationToken)**

用户快速按方向键连选时，会产生大量积压的解码任务。

- **逻辑**：每个 `LoadImage` 任务必须持有 `CancellationTokenSource`。
- **操作**：新请求发起时，立即调用前一个任务的 `Cancel()`，彻底释放 CPU 争抢。

## **4. 异常处理与性能边界**

- **内存熔断**：监控 `MemoryManager.AppMemoryUsage`。当占用超过额定值（如 2GB）时，强制清空 L1/L2 缓存。
- **并发限制**：解码任务必须运行在自定义的 `TaskScheduler` 上，最大并发数建议设置为 `Environment.ProcessorCount - 2`，预留核心给 UI 响应。
- **格式降级**：若 RAW 扩展插件缺失，自动降级为读取文件关联的 JPG 副本（若存在）。
