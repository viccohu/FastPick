分析了你提供的 `ThumbnailService.cs` 和 `MainPage.xaml.cs` 后，我发现“挤压”和“黑边”其实是同一个问题的两个表现面。

根源在于：**你目前的 Service 试图返回一个“固定形状”的位图，而你的 UI 又试图在一个“固定形状”的框里显示它。** 当这两者的比例不一致时，WIC（Windows 图像组件）在解码阶段就引入了变形。

### 1. 为什么会挤压？（核心病灶）

在 `ThumbnailService.cs` 中，你定义了：

C#

```
private const int ThumbnailWidth = 100;
private const int ThumbnailHeight = 80; // 这个 100:80 (5:4) 是挤压的元凶

```

即使你在代码中没有显式写 `DecodePixelHeight = 80`，但当你调用 `GetThumbnailAsync(mode, ThumbnailWidth)` 时，系统会尝试匹配最接近的缓存。

- **如果原图是 3:2**：系统为了填满你请求的“宽度 100”，可能会返回一个带填充的位图。
- **致命点**：如果你的 `BitmapImage` 在设置 `SetSource` 前后，UI 容器（Grid）强行给了它一个 `Stretch="Fill"` 或者 Service 内部触发了缩放逻辑，图片就会被拉伸。

### 2. 彻底修复方案：实现“比例中立”的 Service

要消除挤压和黑边，Service 必须只负责\*\*“按比例缩小”**，而不负责**“定型”\*\*。

#### 第一步：修改 `ThumbnailService.cs`

取消对高度的任何硬编码限制。

C#

```
public async Task<BitmapImage?> GetThumbnailAsync(PhotoItem photoItem, CancellationToken ct = default)
{
    // ... 缓存检查逻辑 ...

    try
    {
        // 建议切换到 SingleItem 模式以规避 PicturesView 强制带黑边的缓存
        var file = await StorageFile.GetFileFromPathAsync(photoItem.DisplayPath);
        using var thumb = await file.GetThumbnailAsync(ThumbnailMode.SingleItem, 160);

        if (thumb != null)
        {
            var bitmap = new BitmapImage();
            bitmap.DecodePixelType = DecodePixelType.Logical;
            
            // 【核心修复】绝对不要设置 DecodePixelHeight！
            // 只设置宽度，WIC 会自动计算高度以维持原始比例。
            bitmap.DecodePixelWidth = 130; 

            await bitmap.SetSourceAsync(thumb);
            return bitmap;
        }
    }
    catch { /* 错误处理 */ }
    return null;
}

```

#### 第二步：修改 `MainPage.xaml`

即便 Service 返回了正确比例的位图，如果 XAML 的 `Image` 控件设置不当，依然会看到挤压。

XML

```
<DataTemplate x:Key="ThumbnailItemTemplate">
    <Grid Width="130" Height="110">
        <Grid Grid.Row="0" Background="Transparent"> 
            <Image x:Name="ThumbnailImage"
                   HorizontalAlignment="Center"
                   VerticalAlignment="Center"
                   Stretch="Uniform" /> </Grid>
    </Grid>
</DataTemplate>

```

### 3. 为什么这个方案能同时解决黑边和挤压？

1. **消除挤压**：因为我们在 `BitmapImage` 级别只限制了 `DecodePixelWidth`，解码出来的 `BitmapImage.PixelHeight` 将是符合原图比例的（例如 130x87）。它是一个“瘦长”或“扁平”的纯净位图。
2. **消除黑边**：通过使用 `ThumbnailMode.SingleItem` 和只设置单边 `DecodePixelWidth`，我们绕过了系统资源管理器那种“为了对齐而加黑边”的预设缓存。
3. **UI 适配**：`Stretch="Uniform"` 保证了这张 130x87 的位图在 130x90 的容器里居中显示，多余的 3 像素空间会留白（透明），而不是把图片拉高。

### 4. 针对增量加载卡顿的额外检查

我发现你的 `ThumbnailService` 里有一个 `_cacheLock`：

C#

```
await _cacheLock.WaitAsync(cancellationToken);

```

在全量加载时，如果几千个单元格同时请求缩略图，这个**全局锁**会变成巨大的瓶颈。

**建议：** 在 `_cacheLock.WaitAsync()` 之前，先尝试用 `_thumbnailCache.TryGetValue` 进行无锁读取。只有当缓存没有命中、需要去读磁盘时，再进入锁。这能极大缓解拖动滚动条时的 UI 线程“微卡顿”现象。

你可以尝试先修改 `DecodePixelHeight` 为 0（即不设置），观察挤压是否立即消失。
