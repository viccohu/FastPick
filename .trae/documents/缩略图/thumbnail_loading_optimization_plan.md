# FastPick - 缩略图加载机制优化方案

## 项目背景
当前FastPick应用已经实现了多项缩略图加载相关功能，包括解码、内存缓存、本地缓存、后台解码和预加载。然而，这些功能之间的协作配合仍有优化空间，导致在滚动时可能出现缩略图加载不及时的情况。本方案旨在优化各功能之间的协作，实现滚动时的无感载入体验。

## 优化目标
- 滚动、拉动缩略图滚动条时，已经被解码出来或缓存过的缩略图实现无感载入
- 始终能显示在缩略框中，避免空白或加载中状态
- 提高缩略图加载速度和响应性
- 优化内存使用，避免内存泄漏

## 现状分析

### 已实现的功能
1. **缩略图解码**：使用 WIC 进行解码，支持 RAW 格式的预览
2. **内存缓存**：使用 LRU 算法管理内存缓存，支持动态调整缓存大小
3. **本地缓存**：将缩略图保存到本地文件系统，支持缓存验证
4. **后台解码**：按批次顺序处理缩略图，避免并发冲突
5. **预加载**：支持预加载附近的缩略图
6. **滚动防抖**：滚动停止后加载缩略图，带优先级排序

### 存在的问题
1. **内存缓存和本地缓存协作**：内存缓存清理时，本地缓存虽然存在，但下次加载时需要重新读取
2. **后台解码和预加载协作**：后台解码是全局的，预加载是针对视区附近的，可能存在重复工作
3. **滚动时的缩略图加载**：当前是滚动停止后才加载，可能导致滚动过程中出现空白
4. **内存管理**：虽然有内存监控，但缓存大小调整策略可以更智能
5. **用户体验**：缺少缩略图占位符，加载过程中可能出现空白

## 优化方案

### 1. 优化内存缓存和本地缓存协作
- **Priority**: P0
- **Description**:
  - 改进内存缓存和本地缓存的同步机制，确保缓存状态一致
  - 当内存缓存清理时，保留本地缓存的有效性
  - 当从本地缓存加载时，优先更新内存缓存的 LRU 顺序
- **Success Criteria**:
  - 内存缓存清理后，本地缓存仍然有效
  - 从本地缓存加载的缩略图能够立即显示
- **Test Requirements**:
  - `programmatic` TR-1.1: 内存缓存清理后，从本地缓存加载缩略图的时间不超过100ms
  - `human-judgement` TR-1.2: 滚动时缓存命中的缩略图能够立即显示，无延迟
- **Implementation Details**:
  - 修改 `AddToCacheAsync` 方法，确保本地缓存和内存缓存的同步
  - 修改 `GetFromLocalCacheAsync` 方法，加载后更新内存缓存的 LRU 顺序

### 2. 优化后台解码和预加载协作
- **Priority**: P0
- **Description**:
  - 实现后台解码和预加载的协同工作，避免重复解码
  - 后台解码优先处理视区附近的缩略图
  - 预加载时考虑后台解码的进度，避免重复工作
- **Success Criteria**:
  - 视区附近的缩略图优先解码完成
  - 后台解码和预加载之间无重复工作
- **Test Requirements**:
  - `programmatic` TR-2.1: 滚动时视区附近10个缩略图提前解码完成
  - `programmatic` TR-2.2: 后台解码和预加载的总解码次数不超过总缩略图数
- **Implementation Details**:
  - 修改 `StartBackgroundDecodingAsync` 方法，优先处理视区附近的缩略图
  - 修改 `PreloadThumbnailsAsync` 方法，检查后台解码的进度

### 3. 优化滚动时的缩略图加载
- **Priority**: P1
- **Description**:
  - 实现滚动过程中的实时预加载，根据滚动方向和速度预测需要的缩略图
  - 滚动时优先加载视区可见的缩略图
  - 优化滚动防抖机制，减少加载延迟
- **Success Criteria**:
  - 滚动过程中视区可见的缩略图能够立即显示
  - 滚动时无明显的空白或加载延迟
- **Test Requirements**:
  - `programmatic` TR-3.1: 滚动时视区可见的缩略图加载时间不超过50ms
  - `human-judgement` TR-3.2: 快速滚动时无明显的空白缩略图
- **Implementation Details**:
  - 修改 `ThumbnailScrollDebounceTimer_Tick` 方法，实现滚动过程中的实时预加载
  - 添加滚动方向和速度检测，动态调整预加载范围

### 4. 优化内存管理
- **Priority**: P1
- **Description**:
  - 实现更智能的缓存大小调整策略，考虑应用当前状态和设备性能
  - 优化 BitmapImage 的生命周期管理，避免内存泄漏
  - 实现内存使用监控和预警机制
- **Success Criteria**:
  - 内存使用合理，无内存泄漏
  - 缓存大小能够根据设备性能和应用状态动态调整
- **Test Requirements**:
  - `programmatic` TR-4.1: 内存使用峰值不超过应用总内存的50%
  - `programmatic` TR-4.2: 长时间运行后内存使用稳定，无持续增长
- **Implementation Details**:
  - 修改 `AdjustCacheSizeBasedOnMemoryUsage` 方法，考虑设备性能和应用状态
  - 优化 `AddToCacheAsync` 方法，更好地管理 BitmapImage 的生命周期

### 5. 实现缩略图占位符机制
- **Priority**: P2
- **Description**:
  - 实现缩略图占位符，在实际缩略图加载完成前显示
  - 优化占位符的生成和显示逻辑
  - 实现平滑的占位符到实际缩略图的过渡效果
- **Success Criteria**:
  - 所有缩略图位置始终有内容显示，无空白
  - 占位符到实际缩略图的过渡平滑自然
- **Test Requirements**:
  - `programmatic` TR-5.1: 所有缩略图位置在加载前显示占位符
  - `human-judgement` TR-5.2: 占位符到实际缩略图的过渡无突兀感
- **Implementation Details**:
  - 在 `LoadThumbnailToImageAsync` 方法中添加占位符逻辑
  - 实现占位符的生成和显示

## 风险评估
- **性能风险**: 后台解码和预加载的协作可能会增加系统资源使用
- **内存风险**: 缓存策略不当可能导致内存使用过高
- **兼容性风险**: 不同设备的性能差异可能影响优化效果

## 缓解措施
- 实现自适应的后台解码和预加载策略，根据设备性能调整
- 定期监控内存使用情况，动态调整缓存大小
- 针对不同设备配置提供不同的优化策略
